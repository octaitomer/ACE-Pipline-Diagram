Phase 3 — Digital Twin & RL Pipeline
eplus_cfg.yaml from ACE Discovery Module 2 is the primary input to SimFarm and the EnergyPlus model
EnergyPlus Model
Physics-based simulation of the data centre. Represents thermal zones, mechanical plant, actuators, and IT load. Foundation for surrogate training.
INeplus_cfg.yaml, weather file (.epw), site drawings
OUTSynthetic simulation data covering full range of operating conditions
EnergyPlus simulates the real DC environment so the RL agent can learn without risk to real infrastructure.

Critical assumption: EnergyPlus provides an accurate, unbiased distribution of transition dynamics. This is validated through Sim2Real tests before go-live.

Plausibility checks at this stage:
  • Annual PUE consistent with site expectation
  • Sensible response to IT load and outdoor temperature changes
  • No unrealistic energy spikes or simulation errors
SimFarm — Surrogate Model
Uses EnergyPlus outputs to train a fast ML-based surrogate. Enables rapid RL training at scale without running the full EnergyPlus simulator on every step.
INEnergyPlus simulation data (eplus_cfg.yaml-driven runs)
OUTFast ML surrogate model of the data centre thermal system
Purpose: Replace the slow physics simulation with a fast ML model for RL training loops.

May use graph neural networks or other ML approaches trained on physics-based simulation data and/or operational data.

Surrogate also used for:
  • Generating savings estimate for customer pricing discussion
  • Running the VM farm simulation environment for Solution Assembly
↓
SynerGym — RL Training Environment
Wraps the surrogate model as a standardised gym environment. The RL agent interacts with it via actions and reward signals to learn an optimal cooling policy.
INSurrogate model, actuator config (from ACE Discovery 2.3), reward function
OUTTrained RL policy (agent checkpoints)
SynerGym is the RL training environment — not just a data translator.

Training validates:
  • Agent policy is sensible (no extreme or unrealistic actions)
  • Thermal SLA is maintained in simulation
  • Implied energy saving ≥10% vs customer baseline
Safe agent tests before go-live:
  • Sim2Real alignment (simulated vs real response times)
  • Safe guardrail behaviour under constraint conflicts
RL Agent (Optimiser)
Trained policy that recommends cooling set-point changes to minimise energy while maintaining thermal SLA. Target: ≥10% cooling energy reduction.
INLive sensor data: temperatures, humidity, IT load, weather
OUTSet-point recommendations (supply air temp, fan speed, chilled water temp)
The Optimiser may be built using RL or other optimisation methodologies. It is trained using the Digital Twin (EnergyPlus + surrogate) as its simulation environment.

Actuators controlled (examples):
  • Cooling unit supply air temperature setpoints
  • Fan speed / airflow rate
  • Chilled water supply temperature
  • Cooling mode selection (free cooling / mechanical)
Note: The accuracy of the Digital Twin directly determines the safety of the Optimiser's recommendations.

Readiness Gate 1 — Deployable?
Structural completeness · Plausibility checks passed · Optimiser trained with sensible policy (≥10% implied saving) · SecOps sign-off
What "Deployable" means: the model can be deployed to site for Ghost Mode — it does NOT authorise live recommendations.

Required evidence:
  • All thermal zones, plant, and actuators represented
  • PUE within expected site range
  • Model responds correctly to IT load and weather changes
  • Trained Optimiser policy is sensible and implies ≥10% savings
  • Digital Twin documentation report produced
  • SecOps/Technical Readiness Check passed (Go/No-Go decision)
Source: Digital Twin Model Readiness Definitions (CAAS space)

Phase 4 — Deployment Infrastructure
ACE Launchpad
Deployment and configuration platform. Pre-populated from ACE Discovery Module 2 data — minimal manual intervention needed.
INACE Discovery Module 2 payload (config.json, HVAC config, actuator config, integration credentials)
OUTConfigured Docker Compose deployment package, secrets provisioned
ACE Launchpad is the single deployment control plane. Module 2 data maps directly to:
  • configuration.json seed values (hall geometry, rack density)
  • Docker Compose env vars (cooling unit parameters, COP, fan config)
  • RL agent config files (actuator bounds, observation nodes)
  • Integration service config (BMS/DCIM endpoint, protocol, polling interval, write-back flag)
  • aceblob:// assets (SimFarm model inputs)
Edge VM (On-site)
Local virtual machine deployed at the customer data centre. Runs all ACE containers including the RL agent, Guardian, and data ingestion services.
INDocker deployment package from Launchpad, network access to BMS/DCIM
OUTRunning ACE service, live sensor ingestion, actuation capability
Why on-site (edge)? All data processing happens at the customer site. No operational data leaves the facility — critical for security and data sovereignty.

Containers running on Edge VM:
  • RL agent (Optimiser)
  • Guardian (safety layer)
  • Data ingestion services (BMS/DCIM connectors)
  • Digital Twin model
  • OctaiPipe Platform monitoring
BMS / DCIM Integration Key dependency
Edge VM connects to the customer's Building Management System (BMS) and/or DCIM. Data ingestion validated. Write-back capability tested and confirmed before any live operation.
INIntegration config from ACE Discovery Module 2.4 (protocol, API endpoint, polling interval)
OUTLive sensor data stream → Edge VM; set-point commands → BMS/DCIM actuators
Data flowing IN from BMS/DCIM:
  • Supply/return air temperatures per zone
  • IT load (per row / hall / total)
  • Cooling unit status, fan speed, setpoints
  • Chilled water temperatures and flow rates
  • Energy metering (HVAC sub-systems)
Commands flowing OUT (when AI ON):
  • Supply air temperature setpoints
  • Fan speed setpoints
  • Chilled water temperature setpoints
Write-back is flagged and tested during ACE Discovery Module 2.4 and Module 3.

Phase 5 — Live Operation (Activation Lifecycle)
Ghost Mode AI OFF
AI agent runs in parallel — observing only. Digital Twin predictions compared against real operational data. Minimum 2 weeks. No set-point changes made.
INLive sensor data from BMS/DCIM
OUTCalibration observations, model error statistics, divergence flags
Ghost Mode is mandatory and safety-critical. The Digital Twin must be proven accurate against real data before any live recommendations are made.

During Ghost Mode:
  • Customer given view-only access to OctaiPipe app
  • Digital Twin tuned on live operational data
  • If building model needs updating → redeployed from VM farm
  • RL agent re-trained on updated Digital Twin
Gate to exit: Readiness Gate 2 (Well-Calibrated) must be passed.
Close Monitoring AI ON
Agent activated with human in the loop. Guardian validates every recommendation before it reaches the BMS. Automation rules agreed and tuned iteratively.
INWell-Calibrated model, agreed automation rules
OUTValidated set-point changes to BMS, confirmed automation rules
Duration: Minimum 2 weeks of close monitoring before full autonomous operation.

During Close Monitoring:
  • Human reviews and approves each recommendation batch
  • Automation rules adjusted as needed
  • Guardian system actively filtering unsafe actions
  • Customer training and enablement sessions
  • Batched learning from real feedback or simulation
Go/No-Go decision taken at end of this phase before full autonomy.
Full Operation AI ON
Autonomous cooling optimisation. Continuous learning from live data. Pilot results reviewed, savings validated, MSA updated for full commercial contract.
INConfirmed automation rules, validated savings baseline
OUTEnergy savings, PUE improvement, federated learnings to base model
Post go-live activities:
  • Continuous model calibration as conditions evolve
  • Federate results to OctaiPipe base model (1 month after go-live)
  • Pilot review: validate savings vs agreed targets
  • If targets met → MSA updated, full contract commences
  • If targets not met → return to agent fine-tuning loop
Expansion: Additional halls/rooms begin a new Activation Project via ACE Discovery Activation Lifecycle.
Readiness Gate 2 — Well-Calibrated?
Authorises exit from Ghost Mode into Close Monitoring. Quantitative thresholds must be met and calibration report approved.
Energy calibration thresholds (ASHRAE Guideline 14 / IPMVP):
  • NMBE: ±5% monthly, ±10% hourly
  • CV(RMSE): ≤15% monthly, ≤20% hourly (IPMVP strict)
  • R² ≥ 0.75 at hourly resolution
Thermal calibration thresholds:
  • Zone/Return air RMSE ≤ 1.5°C
  • Supply air RMSE ≤ 1.0°C
  • Relative Humidity RMSE ≤ 8.0% RH
  • Directionality ≥ 70% at hourly resolution
Required evidence: Full calibration report reviewed and approved by Building Engineer.
Guardian System — Safety Layer Always active once AI ON
Sits between the RL agent and the BMS at all times. Rejects or flags any recommendation that would breach thermal SLA, exceed actuator bounds, or fall outside the model's calibrated operating envelope.
INRL agent set-point recommendations, current sensor state, SLA bounds
OUTApproved commands → BMS/DCIM; rejected commands → operator alert or auto-rollback
Constraint priority (highest first):
  1. Safety (thermal SLA, equipment limits)
  2. Comfort (temperature stability, tenant expectations)
  3. Efficiency (energy minimisation)
Guardian rejects a recommendation if:
  • Action would breach thermal SLA with uncertainty-adjusted probability
  • Action exceeds configured actuator bounds
  • System is operating outside the calibrated envelope
  • Communication failure or sensor anomaly detected
Reference: Jira OP-297 — Guardian rejects recommendations with chance to breach SLA after uncertainty
